Domain 1 — Describe Cloud Concepts 25–30%

Cloud benefits, the shared responsibility model, service types (IaaS/PaaS/SaaS) and deployment models (public/private/hybrid).

What is cloud computing?

  • Cloud computing = delivery of computing services (compute, storage, networking, databases, AI) over the internet, pay-as-you-go.
  • Compute power – CPU/GPU to run apps. Storage – files, databases, blobs.
  • Removes the need to buy and maintain your own physical hardware.

CapEx vs OpEx

CapEx — Capital Expenditure

  • Upfront spend on physical infrastructure (servers, datacentres).
  • Value depreciates over time; you plan years ahead.
  • Traditional on-premises model.

OpEx — Operational Expenditure

  • Pay for services as you consume them (no upfront cost).
  • The cloud / consumption-based model.
  • Deduct from taxes the same year; scale up/down freely.
Exam tip: Cloud = OpEx + consumption-based pricing. "Pay only for what you use" and "no upfront cost" point to the cloud.

Benefits of cloud computing

BenefitMeaning
High availabilityServices stay up via SLAs & redundant infrastructure.
ScalabilityAdd resources to meet demand — vertical (bigger VM) or horizontal (more VMs).
ElasticityAutomatically scale resources up/down with demand (autoscaling).
ReliabilityRecover from failures; distributed across regions.
PredictabilityPerformance (autoscale, load balancing) & cost (TCO calculator, Cost Management).

ManageabilityManage of the cloud (autoscale, templates) & in the cloud (portal, CLI, APIs).
Vertical = scale up (more CPU/RAM on one machine). Horizontal = scale out (more instances). Elasticity = the automatic version of scaling.

Shared Responsibility Model

As you move IaaS → PaaS → SaaS, the provider takes on more responsibility. You always keep: your data, accounts/identities, and devices.
  • Always the customer's responsibility: data, information, account & access management, endpoints/devices.
  • Always the provider's responsibility: physical hosts, physical network, physical datacentre.
  • Varies by service type: OS, network controls, applications, identity & directory infrastructure.

Cloud Service Types

TypeYou manageExamples / use
IaaS (Infrastructure)Most control — OS, runtime, apps, data. Provider handles hardware.Azure VMs, lift-and-shift, full control, test/dev.
PaaS (Platform)Just your app & data. Provider handles OS, middleware, runtime.Azure App Service, SQL Database — dev focus.
SaaS (Software)Just data & access. Provider handles everything else.Microsoft 365, Outlook, Dynamics 365.
Exam tip: "Maximum control / lift-and-shift" = IaaS. "Developers build apps without managing OS" = PaaS. "Ready-to-use software, just log in" = SaaS.

Cloud Deployment Models

Public

  • Owned by a cloud provider, shared by all.
  • No CapEx, high scale, pay-as-you-go.
  • Less control over security.

Private

  • Used by a single organisation.
  • More control & compliance; higher cost.
  • Can be on-prem or hosted.

Hybrid

  • Combines public + private.
  • Flexibility, keep sensitive data private.
  • Most complex to manage.
Multi-cloud = using more than one provider (e.g. Azure + AWS). Azure Arc manages resources across on-prem, multi-cloud and edge. Azure VMware Solution runs VMware workloads natively in Azure.

The Consumption-Based Model & Pricing Models

  • No upfront cost; pay only for resources used; stop paying when you stop using.
  • Benefits: better cash-flow planning, scale precisely with demand, no wasted idle capacity.
  • Pay-as-you-go — pay per unit consumed, no commitment (most flexible, highest unit price).
  • Reserved / commitment — commit 1 or 3 years for a large discount on predictable workloads.
  • Spot — bid on spare capacity at deep discount; can be evicted when Azure needs it.
  • Hybrid Benefit / BYOL — reuse existing licences to lower cost.

Serverless

  • Serverless = run code/services without provisioning or managing any servers; the platform handles all infrastructure.
  • Event-driven & auto-scaling — scales from zero to thousands automatically.
  • Pay per execution — billed only while your code runs, not for idle time.
  • Examples: Azure Functions (code), Logic Apps (workflows), Azure Container Instances, serverless SQL/Cosmos DB.
Exam tip: "No servers to manage + pay only when code runs + scales automatically" = serverless. The flagship example is Azure Functions.

Domain 2 — Azure Architecture & Services 35–40%

The biggest domain: physical/management hierarchy, core compute, networking and storage services, plus identity, access & security.

Core Architectural Components

Geography → Region → Availability Zone → Datacentre. Region pairs sit ≥300 miles apart for resilience.
  • Region: a geographical area containing one or more datacentres connected by a low-latency network.
  • Availability Zone (AZ): physically separate datacentres within a region — own power/cooling/network. Protects against datacentre failure. Use ≥2 for high availability.
  • Region pair: two regions ≥300 miles apart; updates roll out one at a time; many services replicate here automatically.
  • Sovereign regions: isolated instances — Azure Government (US) and Azure China (operated by 21Vianet).

Management Hierarchy (scope of governance)

Policies & RBAC applied at a higher level are inherited downward.
  • Resources: the services you create (VMs, databases, storage accounts…).
  • Resource Group: logical container. A resource lives in exactly one RG; delete the RG → delete all its resources.
  • Subscription: unit of billing & access; logically groups resource groups. Has spending limits/quotas.
  • Management Group: container above subscriptions to apply policy & access at scale (inherited downward).

Exam tip: Containers are lightweight & portable (share host OS) vs VMs (each has own OS). Functions = serverless/event-driven. AKS = orchestrate many containers.

Resources required for a VM

  • A VM doesn't stand alone — creating one provisions: size (CPU/RAM family), OS disk + optional data disks (managed disks), a virtual network & subnet, a network interface (NIC), optionally a public IP, and a network security group (NSG).

Application hosting options

  • VMs (IaaS, full control) → Containers (ACI/AKS, portable, microservices) → Web Apps / App Service (PaaS, just deploy code). Choose based on how much control vs how little management you want.

Blob access tiers (cost vs access frequency)

  • Hot – frequently accessed; highest storage cost, lowest access cost.
  • Cool – infrequent, stored ≥30 days; lower storage cost.
  • Cold – rarely accessed, ≥90 days; cheaper still.
  • Archive – rarely accessed, ≥180 days; cheapest storage, highest retrieval cost & latency (offline).

Identity, Access & Security

Note: in the current syllabus this sits under Azure architecture & services, not management & governance.

  • Microsoft Entra ID (formerly Azure AD) – cloud identity & access management; sign-in & directory service.
  • Authentication (AuthN) = proving who you are. Authorization (AuthZ) = what you're allowed to do.
  • Single Sign-On (SSO) – one set of credentials for many apps.
  • Multi-Factor Authentication (MFA) – two+ of: something you know (password), have (phone/token), are (biometrics).
  • Passwordless authentication – sign in without a password using Windows Hello, the Microsoft Authenticator app, or a FIDO2 security key. More secure & convenient than passwords.
  • Conditional Access – grant/block/require MFA based on signals (location, device, risk).
  • Entra Domain Services – managed domain services (join, group policy, LDAP) without managing domain controllers.
  • External identities – B2B (partners) & B2C (customers).

Zero Trust & Defence in Depth

Zero Trust

  • "Never trust, always verify."
  • Verify explicitly, least-privilege access, assume breach.

Defence in Depth

  • Layered security; if one layer fails, others protect.
  • Layers: physical → identity → perimeter → network → compute → app → data.

Access control & security tools

  • Azure RBAC (Role-Based Access Control) – assign roles (Owner, Contributor, Reader) at a scope (MG/sub/RG/resource); least-privilege.
  • Microsoft Defender for Cloud – security posture management & threat protection; secure score.
  • Azure DDoS Protection, Azure Firewall, Key Vault (store secrets/keys/certs).
Exam tip: Entra ID = identity. RBAC = what a user can do to Azure resources. Defender for Cloud = security monitoring/score. Conditional Access = sign-in policies (e.g. require MFA off-network).

Domain 3 — Management & Governance 30–35%

Cost management, governance & compliance, and the tools to deploy & monitor everything. (Identity, access & security is covered in Domain 2 per the current syllabus.)

Cost Management

  • Factors affecting cost: resource type, consumption, region, network traffic (egress/bandwidth), billing zone.
  • Azure Pricing Calculator – estimate cost of a planned solution before deploying.
  • TCO (Total Cost of Ownership) Calculator – compare on-premises cost vs Azure (shows savings). (Useful background — no longer an explicit exam objective.)
  • Microsoft Cost Management – analyse, monitor & control actual spend; set budgets & alerts.
  • Tags – metadata (key:value) on resources to organise & track cost by department/project/environment.

Ways to reduce cost

  • Reservations (reserved instances) – commit 1 or 3 years for big discounts.
  • Azure Hybrid Benefit – reuse existing Windows Server / SQL licences.
  • Spot VMs – use spare capacity at deep discount (can be evicted).
  • Right-size, autoscale, shut down idle resources, choose cheaper regions.
Exam tip: Pricing Calculator = estimate a new solution. TCO Calculator = compare on-prem vs cloud. Cost Management = track/limit money already being spent.

Service Level Agreements (SLAs)

  • SLA = Microsoft's commitment to uptime & connectivity, expressed as a % (e.g. 99.9%).
  • Higher 9s = less downtime: 99.9% ≈ 8.76 h/yr down; 99.99% ≈ 52 min/yr.
  • Composite SLA: multiply service SLAs (chaining services lowers the combined SLA).
  • Service Credits: if Microsoft misses the SLA, you may get billing credits.
  • Free/Preview services generally have no SLA. Adding AZs/redundancy raises effective availability.

One-Page Cheat Sheet

Pick the service type

  • Full control / lift-and-shift → IaaS (VM)
  • Build app, no OS hassle → PaaS (App Service)
  • Just use software → SaaS (M365)
  • Event-driven, serverless → Functions
  • Many containers at scale → AKS

Pick the cost tool

  • Estimate a new build → Pricing Calculator
  • On-prem vs cloud comparison → TCO Calculator
  • Track/limit live spend → Cost Management
  • Discount for 1–3 yr commit → Reservations

Pick the connectivity

  • Private, dedicated, fast → ExpressRoute
  • Encrypted over internet → VPN Gateway
  • VNet ↔ VNet → Peering
  • Filter traffic → NSG

Pick the monitor/govern tool

  • Recommendations → Advisor
  • Azure outages → Service Health
  • Enforce rules → Azure Policy
  • Stop deletion → Resource Lock
  • Security score → Defender for Cloud
